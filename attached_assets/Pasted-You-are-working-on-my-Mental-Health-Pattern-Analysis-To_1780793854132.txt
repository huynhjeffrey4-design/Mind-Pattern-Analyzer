# 1. Input and Data Injection Vulnerability Tests

## 1.1 SQL Injection: Login Form

Test that the login endpoint does not allow SQL injection.

### Test case

Send a login request with SQL injection-style input:

```json
{
  "email": "' OR '1'='1",
  "password": "' OR '1'='1"
}
```

### Expected result

* Login fails.
* Response status should be `401`, `400`, or the app’s existing invalid-login status.
* No access token is returned.

---

## 1.2 SQL Injection: Journal Entry Content

Test that journal content is safely stored as text and does not execute as SQL.

### Test case

Create/authenticate a user.

Submit journal content like:

```text
'); DROP TABLE users; --
```

### Expected result

* The request should not crash the server.
* The journal entry may be rejected by validation or safely stored as text.
* The `users` table must still exist afterward.
* Existing users must still be queryable.
* The app must not execute the injected SQL.

---

### Test case


### Expected result

* The request should not crash the server.
* The check-in may be rejected by validation or safely stored as text.

---

## 1.4 XSS: Journal Entry Content

Test that malicious script input does not execute or become unsafe output.

### Test case

Create/authenticate a user.

Submit journal content:

```html
<script>alert('xss')</script>
```

Then fetch the journal entry.

### Expected result

* The server should not execute anything.
* The API should return JSON only.
* The content should either:

  * be safely escaped/sanitized, or
  * be returned as plain text for the frontend to safely render.
* The response content type should be JSON.
* The test should document the current expected behavior clearly.

Important: Do not use a real browser exploit. This is a defensive unit/integration test.

---

## 1.5 XSS: Check-In Notes

Test that script-like input in check-in notes does not break the API.

### Test case

Create/authenticate a user.

Submit check-in notes:

```html
<img src=x onerror=alert('xss')>
```

Then fetch the check-in.

### Expected result

* API returns JSON.
* Server does not crash.
* Input is either escaped/sanitized or safely returned as plain text.
* No HTML page is rendered from user input.

Check whether the app currently has any file upload endpoint.

Expected result:

# 2. Authentication and Authorization Flaw Tests

## 2.1 Weak Password Rejection

Test that registration rejects weak passwords.

### Test cases

Attempt to register with:

```text
123
password
abc
```

### Expected result

* Registration should fail with `400` or `422`.
* User should not be created.

If strong password validation is not implemented yet, add the smallest validation needed, such as minimum length 8.

Do not add complicated password rules unless already part of the app’s design.

---

## 2.2 Brute Force / Credential Stuffing Basic Protection

Test current behavior for repeated failed login attempts.

### Test case

Attempt to log in to the same account with the wrong password multiple times, for example 5 to 10 failed attempts.

### Expected result

Preferred:

* App returns `429 Too Many Requests` after repeated failures if rate limiting exists.

If rate limiting is not implemented:

* Add a test marked as expected failure or skipped with a clear reason, OR
* Add a small documented TODO test explaining that login rate limiting is not yet implemented.

Do not implement a large rate-limiting system unless the current project already has a pattern for it.

## 2.4 CSRF Protection Check

Since the app uses JWT bearer tokens for API authentication, CSRF risk depends on where the token is stored and how requests are authenticated.

### Test case

Try to call protected state-changing endpoints without the Authorization header:

* `POST /api/checkins`
* `PUT /api/checkins/{id}`
* `DELETE /api/journals/{id}`
* `PUT /api/users/me/settings`

### Expected result

* All should return `401` or the app’s existing unauthorized status.
* No resource should be created, updated, or deleted without a valid bearer token.

---

## 2.5 Broken Access Control: Other User’s Journal

Create User A and User B.

User A creates a journal entry.

Authenticate as User B.

Try:

```text
GET /api/journals/{user_a_journal_id}
DELETE /api/journals/{user_a_journal_id}
```

### Expected result

* User B cannot view User A’s journal.
* User B cannot delete User A’s journal.
* Expected status: `403` or `404`.
* User A’s journal still exists afterward.

---

## 2.7 Broken Access Control: Other User’s Settings

Create User A and User B.

Authenticate as User B.

Attempt to access or modify User A’s settings through any guessed URL or ID-based endpoint, if one exists.

### Expected result

* The app should only expose current-user settings through `/api/users/me/settings`.
* User B should not be able to access User A’s settings by changing a URL ID.
* If no ID-based settings endpoint exists, document that this is good.

## 2.8 Unauthenticated URL Access to Private Data

Test that a user cannot access private data through a direct URL when not logged in.

### Test protected endpoints without a token

Test:

```text
GET /api/checkins
GET /api/checkins/{checkin_id}
POST /api/checkins
PUT /api/checkins/{checkin_id}
GET /api/journals
GET /api/journals/{journal_id}
POST /api/journals
DELETE /api/journals/{journal_id}
GET /api/insights
POST /api/insights/generate
GET /api/users/me/settings
PUT /api/users/me/settings
```

### Expected result

* Every protected endpoint returns `401` or the app’s existing unauthorized status.
* No private data is returned.
* No state is changed.

This is especially important: users must not be able to access someone else’s data by guessing or typing a URL.

# 3. Server Configuration and Encryption Tests

## 3.1 Missing Security Headers

Write tests for HTTP security headers on API responses.

Check for these headers if supported by the app:

```text
X-Content-Type-Options: nosniff
X-Frame-Options: DENY or SAMEORIGIN
Referrer-Policy
Content-Security-Policy
```

### Expected result

Preferred:

* API responses include basic security headers.

If these headers are missing:

* Add the smallest FastAPI middleware needed to include basic security headers.
* Do not overcomplicate CSP. A simple safe default is acceptable for the API.

Example acceptable MVP headers:

```text
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer
Content-Security-Policy: default-src 'self'
```

# 4. User Data Isolation Through URL Tests

This is the most important part.

Write explicit tests proving that users cannot access other users’ private resources by changing IDs in the URL.

## 4.1 Journal URL Isolation

* User A creates journal entry.
* User B logs in.
* User B requests `/api/journals/{user_a_journal_id}`.
* Expected: `403` or `404`.

## 4.2 Journal Delete URL Isolation

* User A creates journal entry.
* User B logs in.
* User B requests `DELETE /api/journals/{user_a_journal_id}`.
* Expected: `403` or `404`.
* User A logs in.
* User A can still fetch their journal.

## 4.3 Check-In URL Isolation

* User A creates check-in.
* User B logs in.
* User B requests `/api/checkins/{user_a_checkin_id}`.
* Expected: `403` or `404`.

## 4.4 Check-In Update URL Isolation

* User A creates check-in.
* User B logs in.
* User B sends `PUT /api/checkins/{user_a_checkin_id}` with new values.
* Expected: `403` or `404`.
* User A logs in.
* User A’s check-in values are unchanged.

## 4.5 Insight URL Isolation

If insight endpoints expose specific insight IDs:

* User A generates insight.
* User B tries to access User A’s insight by ID.
* Expected: `403` or `404`.

If insights are only exposed through `/api/insights`, then:

* User B should only see User B’s insights.
* User B should never see User A’s insights.
