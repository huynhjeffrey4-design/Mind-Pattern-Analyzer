## Task: Write backend tests for daily check-in once-per-day behavior, editing today’s check-in, and viewing daily check-ins

Focus only on tests for the Daily Check-In feature. Do not refactor unrelated files. Do not add unrelated features.

The intended behavior is:

1. A user can create **only one check-in per calendar day**.
2. If the user already has a check-in for today, they should be able to **edit/update today’s check-in** instead of creating a second one.
3. The user should be able to **view their daily check-ins**.
4. Users must only be able to view and edit their own check-ins.

---

## Tests to add

Add backend tests covering these cases:

### 1. User can create one daily check-in

* Create/authenticate a test user.
* Call `POST /api/checkins` with valid check-in data.
* Assert the response status is `200` or `201`, depending on current API behavior.
* Assert the response includes expected fields:

  * `id`
  * `mood_score`
  * `stress_score`
  * `sleep_hours`
  * `exercise`
  * `social_interaction`
  * `workload_score`
  * `energy_score`
  * `notes`
  * `created_at`

### 2. User cannot create two check-ins on the same day

* Create/authenticate a test user.
* Create a check-in for today.
* Attempt to create a second check-in for the same user on the same day.
* Assert the API does **not** create a duplicate daily check-in.
* Expected behavior can be either:

  * return `400` or `409` with a clear error message, or
  * return the existing check-in if that is how the current API is designed.
* After the second request, call `GET /api/checkins`.
* Assert there is still only **one** check-in for today.

### 3. User can edit today’s check-in

* Create/authenticate a test user.
* Create a check-in for today.
* Call the update endpoint for that check-in.
* Use updated values, for example:

  * `mood_score` changes from `3` to `5`
  * `stress_score` changes from `4` to `2`
  * `notes` changes to `"Updated check-in for today."`
* Assert response status is `200`.
* Assert the returned check-in contains the updated values.
* Fetch the check-in again and assert the saved values were changed.

### 4. User can view all of their daily check-ins

* Create/authenticate a test user.
* Create multiple check-ins on different dates.
* Call `GET /api/checkins`.
* Assert the response status is `200`.
* Assert the response is a list.
* Assert all of that user’s check-ins appear.
* Assert each item includes:

  * `id`
  * `mood_score`
  * `stress_score`
  * `sleep_hours`
  * `created_at`

### 5. User can view one daily check-in by ID

* Create/authenticate a test user.
* Create one check-in.
* Call `GET /api/checkins/{checkin_id}`.
* Assert the response status is `200`.
* Assert the returned check-in has the correct `id`.
* Assert the returned check-in has the correct values.

### 6. User isolation for viewing check-ins

* Create User A and User B.
* Create a check-in for User A.
* Authenticate as User B.
* Call `GET /api/checkins/{user_a_checkin_id}`.
* Assert User B cannot view User A’s check-in.
* Expected status should be `404` or `403`, depending on current API design.

### 7. User isolation for editing check-ins

* Create User A and User B.
* Create a check-in for User A.
* Authenticate as User B.
* Try to update User A’s check-in.
* Assert User B cannot update User A’s check-in.
* Expected status should be `404` or `403`.
* Authenticate as User A.
* Fetch the check-in again.
* Assert the original values were not changed by User B.

### 8. Unauthenticated access protection

* Call `GET /api/checkins` without a token.
* Assert it returns `401` or the app’s existing unauthorized status.
* Call `POST /api/checkins` without a token.
* Assert it returns `401` or the app’s existing unauthorized status.
* Call `PUT /api/checkins/{checkin_id}` without a token.
* Assert it returns `401` or the app’s existing unauthorized status.
